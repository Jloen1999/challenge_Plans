1. Bienvenida: Qu es Express y por qu usarlo?

Qu es Express exactamente?

Imagina que quieres construir una casa. Podras empezar desde cero, fabricando cada ladrillo, mezclando el cemento, cortando la madera... o podras usar materiales prefabricados que hacen el trabajo mucho ms rpido! Express.js es como ese conjunto de materiales prefabricados, pero para crear aplicaciones web con JavaScript.

En trminos ms tcnicos, Express es un framework (un marco de trabajo) para Node.js que simplifica la creacin de servidores web y APIs. Es como un ayudante que te proporciona herramientas y estructuras ya preparadas para que no tengas que escribir todo desde cero.

  res.send('Hola! Bienvenido a mi primera app con Express');
});

  console.log('Servidor funcionando en http://localhost:3000');
});

Por qu la gente adora Express?

  1. Es simple y no te complica la vida: Express no te obliga a seguir reglas estrictas. Es como un lienzo en blanco donde t decides cmo organizar tu cdigo.

  2. Es rpido y ligero: No viene con un montn de funciones que quizs nunca uses. Solo incluye lo esencial, y t aades lo que necesites.

  3. Tiene una comunidad enorme: Si te encuentras con un problema, es casi seguro que alguien ya lo ha solucionado antes.

  4. Es muy flexible: Puedes usarlo para crear desde una pequea API hasta una aplicacin web compleja como nuestra plataforma Challenge Plans.

3. Conceptos Fundamentales: Las Piezas del Rompecabezas

La Aplicacin Express

Todo comienza con la creacin de una aplicacin Express. Piensa en ella como el "cerebro" de tu servidor:

Esta app es el objeto principal que usars para configurar todo tu servidor. Algunas cosas que puedes hacer con ella:

  • Definir rutas (app.get(), app.post(), etc.)
  • Configurar middlewares (app.use())
  • Iniciar el servidor (app.listen())

En nuestro proyecto Challenge Plans, configuramos la app en src/app.ts:

// Middleware para manejar errores

Las Peticiones (Request o req)

Cuando alguien visita tu sitio web o usa tu API, enva una peticin (request) a tu servidor. Esta peticin contiene mucha informacin til:

  • req.params: Parmetros en la URL (como /usuarios/123, donde 123 es un parmetro)
  • req.query: Informacin en la query string (como /buscar?nombre=Juan)
  • req.body: Datos enviados en el cuerpo de la peticin (por ejemplo, desde un formulario)
  • req.headers: Cabeceras HTTP con metadatos de la peticin
// Ejemplo: accediendo a la informacin de la peticin
app.get('/saludo/:nombre', (req, res) => {
  const nombre = req.params.nombre;        // Viene de la URL
  const idioma = req.query.idioma || 'es'; // Viene de ?idioma=en
  
  if (idioma === 'en') {
    res.send(`Hello, ${nombre}!`);
  } else {
    res.send(`Hola, ${nombre}!`);
  }
});

Las Respuestas (Response o res)

Despus de procesar una peticin, necesitas enviar una respuesta (response). Hay muchas formas de responder:

  • res.send(): Enva una respuesta bsica (texto, HTML, etc.)
  • res.json(): Enva datos en formato JSON (ideal para APIs)
  • res.render(): Renderiza una vista (para aplicaciones con frontend)
  • res.status(): Establece el cdigo de estado HTTP (200 para xito, 404 para no encontrado, etc.)
// Ejemplo: diferentes tipos de respuesta

  res.status(404).send('Pgina no encontrada');  // Establece cdigo 404

HTTP define varios "verbos" o mtodos para interactuar con recursos:

Parmetros en Rutas: Rutas Dinmicas

Los parmetros en rutas te permiten capturar valores variables de la URL. Se definen con dos puntos : seguidos del nombre del parmetro:

Si alguien visita /productos?categoria=electronica&ordenar=nombre, tendrs:

5. Middlewares: Los Ayudantes de Express

Qu son los Middlewares?

Los middlewares son probablemente el concepto ms importante para entender Express. Un middleware es simplemente una funcin que se ejecuta entre la recepcin de una peticin y el envo de una respuesta.

Imagina una fbrica donde un producto pasa por varias estaciones antes de estar completo. Cada estacin aade o modifica algo. Los middlewares son como esas estaciones para las peticiones HTTP.

// Estructura bsica de un middleware
function miMiddleware(req, res, next) {
  // 1. Hacer algo con la peticin (req)
  // 2. Hacer algo con la respuesta (res)
  // 3. Llamar a next() para pasar al siguiente middleware
  next();
}

// Usar el middleware en toda la aplicacin
app.use(miMiddleware);

El parmetro next es crucial: es una funcin que, cuando se llama, pasa la peticin al siguiente middleware en la cadena.

Middlewares Integrados y Populares

Express incluye algunos middlewares tiles, y hay muchos ms disponibles como paquetes npm:

// Middleware para procesar JSON en las peticiones
app.use(express.json());

// Middleware para procesar datos de formularios
app.use(express.urlencoded({ extended: true }));

// Middleware para servir archivos estticos (imgenes, CSS, etc.)
app.use(express.static('public'));

En Challenge Plans, usamos varios middlewares populares:

// De src/app.ts
app.use(helmet());         // Aade cabeceras de seguridad
app.use(compression());    // Comprime respuestas para mejorar rendimiento
app.use(morgan('dev'));    // Registra peticiones para debugging
app.use(cors());           // Permite peticiones desde otros dominios
app.use(express.json());   // Procesa cuerpos JSON

Creando tu Propio Middleware

Crear tu propio middleware es simple. Aqu hay un ejemplo de un middleware de registro que anota cundo ocurre cada peticin:

// Middleware de registro
function loggerMiddleware(req, res, next) {
  const timestamp = new Date().toISOString();
  console.log(`[${timestamp}] ${req.method} ${req.url}`);
  next(); // No olvides llamar a next()!
}

// Usar el middleware
app.use(loggerMiddleware);

Los middlewares se pueden aplicar:

  • A toda la aplicacin: app.use(middleware)
  • A una ruta especfica: app.use('/api', middleware)
  • A un mtodo HTTP especfico: app.get('/api', middleware, controlador)

Middleware de Autenticacin: Un Ejemplo Prctico

Un uso comn de middlewares es verificar si un usuario est autenticado antes de permitirle acceder a ciertas rutas. En Challenge Plans, usamos JWT (JSON Web Tokens) para autenticacin:

// Versin simplificada de nuestro middleware de autenticacin
export const authenticate = (req, res, next) => {
  try {
    // 1. Obtener el token del header de autorizacin
    const authHeader = req.headers.authorization;
    if (!authHeader) {
      return res.status(401).json({ message: 'No hay token' });
    }
    
    // 2. Extraer el token (viene como "Bearer [token]")
    const token = authHeader.split(' ')[1];
    
    // 3. Verificar el token
    const decodedToken = jwt.verify(token, process.env.JWT_SECRET);
    
    // 4. Aadir la informacin del usuario a la peticin
    req.usuario = decodedToken;
    
    // 5. Continuar con el siguiente middleware o controlador
    next();
  } catch (error) {
    return res.status(401).json({ message: 'Token invlido' });
  }
};

Y luego lo usamos para proteger rutas:

// Rutas pblicas (no necesitan autenticacin)

// Rutas protegidas (requieren autenticacin)
router.use('/retos', authenticate, retosRoutes);
router.use('/planes', authenticate, planesRoutes);

6. Controladores: Donde Sucede la Accin

Qu es un Controlador?

Un controlador es una funcin (o grupo de funciones) que maneja la lgica asociada a una ruta especfica. Los controladores reciben la peticin, procesan los datos necesarios, y envan una respuesta.

Si las rutas son como las direcciones de tu aplicacin, los controladores son como las personas que atienden esas direcciones.

}

// Usar el controlador con una ruta

7. Servicios: Separando la Lgica de Negocio

Por qu Necesitamos Servicios?

Los servicios son una capa adicional que separa la lgica de negocio (las reglas de cmo funciona tu aplicacin) de los controladores (que manejan las peticiones HTTP).

Imagina que tienes una tienda online. Los controladores seran como los vendedores que atienden a los clientes, mientras que los servicios seran como el equipo de almacn que realmente busca los productos, verifica el inventario, etc.

Beneficios de usar servicios:

  1. Reutilizacin: La misma lgica puede usarse en diferentes controladores
  2. Pruebas ms fciles: Puedes probar la lgica sin simular peticiones HTTP
  3. Cdigo ms limpio: Los controladores solo se preocupan de manejar HTTP, no de las reglas de negocio

Ejemplo de un Servicio: RetosService

Veamos un ejemplo simplificado del servicio de retos en Challenge Plans:

// Versin simplificada de services/retos.service.ts
export class RetosService {
  // Repositorio para acceder a la base de datos
  private retoRepository = getRepository(Reto);
  
  // Obtener todos los retos con filtros opcionales
  async obtenerTodos(filtros = {}) {
    try {
      // Construir la consulta segn los filtros
      const consulta = this.buildQuery(filtros);
      
      // Ejecutar la consulta y devolver resultados
      const retos = await this.retoRepository.find(consulta);
      return retos;
    } catch (error) {
      throw new Error(`Error al obtener retos: ${error.message}`);
    }
  }
  
  // Obtener un reto especfico por ID
  async obtenerPorId(id: string) {
    try {
      const reto = await this.retoRepository.findOne({
        where: { id },
        relations: ['tareas', 'categorias'] // Incluir relaciones
      });
      
      if (!reto) {
        throw new Error('Reto no encontrado');
      }
      
      return reto;
    } catch (error) {
      throw new Error(`Error al obtener el reto: ${error.message}`);
    }
  }
  
  // Crear un nuevo reto
  async crear(datosReto) {
    try {
      // Validar datos (podras usar una funcin separada)
      this.validarDatosReto(datosReto);
      
      // Crear entidad y guardar
      const nuevoReto = this.retoRepository.create(datosReto);
      await this.retoRepository.save(nuevoReto);
      return nuevoReto;
    }
  }
  
  // Actualizar un reto existente
  async actualizar(id: string, datosReto) {
    try {
      // Verificar que el reto existe
      const retoExistente = await this.obtenerPorId(id);
      
      // Actualizar los campos con los nuevos datos
      Object.assign(retoExistente, datosReto);
      
      // Guardar los cambios
      const retoActualizado = await this.retoRepository.save(retoExistente);
      return retoActualizado;
    } catch (error) {
      throw new Error(`Error al actualizar el reto: ${error.message}`);
    }
  }
  
  // Eliminar un reto
  async eliminar(id: string) {
    try {
      // Verificar que el reto existe
      const retoExistente = await this.obtenerPorId(id);
      
      // Eliminar el reto
      await this.retoRepository.remove(retoExistente);
      return { success: true, message: 'Reto eliminado correctamente' };
    } catch (error) {
      throw new Error(`Error al eliminar el reto: ${error.message}`);
    }
  }
}

8. Manejo de Errores: Cuando las Cosas Salen Mal

Errores en Express

En toda aplicacin, las cosas pueden salir mal. Una base de datos puede estar cada, un usuario puede enviar datos incorrectos, o simplemente podemos tener un error en nuestro cdigo. Express facilita el manejo centralizado de estos errores.

En Express, los errores se manejan principalmente de dos formas:

  1. Try/Catch en controladores: Para capturar errores en funciones asncronas
  2. Middleware de errores: Para procesar todos los errores de manera centralizada
// Ejemplo de try/catch en un controlador
app.get('/usuarios/:id', async (req, res, next) => {
  try {
    // Esto podra fallar (por ejemplo, si la base de datos est cada)
    const usuario = await obtenerUsuario(req.params.id);
    
    // Si el usuario no existe, respondemos con un 404
    if (!usuario) {
      return res.status(404).send('Usuario no encontrado');
    }
    
    // Si todo va bien, enviamos el usuario
    res.json(usuario);
  } catch (error) {
    // Si ocurre un error, lo pasamos al middleware de errores
    next(error);
  }
});

Middleware de Errores Personalizado

El middleware de errores es como un "atrapador" que captura cualquier error que ocurra durante el procesamiento de una peticin. Se define con 4 parmetros (en lugar de los 3 habituales):

// Middleware de errores bsico
app.use((error, req, res, next) => {
  console.error('Error:', error);
  
  res.status(500).json({
    mensaje: 'Algo sali mal',
    error: process.env.NODE_ENV === 'development' ? error.message : {}
  });
});

En nuestro proyecto, usamos un middleware de errores ms sofisticado que maneja diferentes tipos de errores:

// middlewares/error.middleware.ts
export const errorMiddleware = (error, req, res, next) => {
  // Registrar el error para debugging interno
  console.error(`[ERROR] ${req.method} ${req.path}:`, error);
  
  // Diferentes respuestas segn el tipo de error
  if (error instanceof NotFoundError) {
    // Errores 404 - Recurso no encontrado
    return res.status(404).json({
      status: 'error',
      message: error.message
    });
  }
  
  if (error instanceof ValidationError) {
    // Errores 400 - Datos invlidos
    return res.status(400).json({
      status: 'error',
      message: 'Datos de entrada invlidos',
      errors: error.errors
    });
  }
  
  if (error instanceof UnauthorizedError) {
    // Errores 401 - No autenticado o no autorizado
    return res.status(401).json({
      status: 'error',
      message: error.message
    });
  }
  
  // Error genrico para cualquier otro caso
  res.status(500).json({
    status: 'error',
    message: 'Error interno del servidor',
    // Solo mostrar detalles tcnicos en desarrollo
    details: process.env.NODE_ENV === 'development' ? error.message : undefined
  });
};

Clases de Error Personalizadas

Para facilitar el manejo consistente de errores, es til definir clases de error personalizadas:

// utils/custom-errors.ts

// Error base para todos nuestros errores personalizados
export class CustomError extends Error {
  statusCode: number;
  errors?: any[];
  
  constructor(message: string, statusCode: number = 500, errors?: any[]) {
    super(message);
    this.statusCode = statusCode;
    this.errors = errors;
    Object.setPrototypeOf(this, CustomError.prototype);
  }
}

// Error para recursos no encontrados (404)
export class NotFoundError extends CustomError {
  constructor(resource: string = 'Recurso') {
    super(`${resource} no encontrado`, 404);
    Object.setPrototypeOf(this, NotFoundError.prototype);
  }
}

// Error para problemas de autenticacin/autorizacin (401)
export class UnauthorizedError extends CustomError {
  constructor(message: string = 'No ests autorizado para realizar esta accin') {
    super(message, 401);
    Object.setPrototypeOf(this, UnauthorizedError.prototype);
  }
}

// Error para datos de entrada invlidos (400)
export class ValidationError extends CustomError {
  constructor(errors: any[]) {
    super('Error de validacin', 400, errors);
    Object.setPrototypeOf(this, ValidationError.prototype);
  }
}

9. Conectando con la Base de Datos

TypeORM: Un Puente Entre Express y la Base de Datos

En nuestro proyecto Challenge Plans, usamos TypeORM para interactuar con la base de datos PostgreSQL. TypeORM es un ORM (Object-Relational Mapping) que nos permite trabajar con la base de datos usando clases y objetos en lugar de SQL puro.

Imagina TypeORM como un traductor: nosotros hablamos en "JavaScript/TypeScript", y l traduce nuestras instrucciones a SQL para que la base de datos las entienda.

Las ventajas de usar TypeORM son:

  1. Cdigo ms limpio: Trabajamos con objetos en lugar de SQL
  2. Mayor seguridad: Previene ataques de inyeccin SQL
  3. Independencia del motor de BD: Podemos cambiar de PostgreSQL a MySQL sin cambiar el cdigo
  4. Migraciones automticas: Genera y ejecuta scripts para actualizar la estructura de la BD

10. Autenticacin y Seguridad

Autenticacin con JWT

Para proteger nuestras rutas en Challenge Plans, utilizamos JSON Web Tokens (JWT). Los JWT son como "tarjetas de identificacin digitales" que el servidor emite cuando un usuario se autentica correctamente.

El flujo de autenticacin funciona as:

  1. El usuario enva sus credenciales (email y contrasea)
  2. El servidor verifica las credenciales en la base de datos
  3. Si son correctas, el servidor crea un token JWT y lo enva al cliente
  4. El cliente guarda este token y lo enva en cada solicitud posterior
  5. El servidor verifica el token para identificar al usuario

Veamos cmo implementamos esto:

// services/auth.service.ts
import jwt from 'jsonwebtoken';
import bcrypt from 'bcrypt';

export class AuthService {
  private userRepository = AppDataSource.getRepository(Usuario);
  
  // Mtodo para autenticar usuario
  async login(email: string, password: string) {
    // Buscar el usuario por email
    const usuario = await this.userRepository.findOne({ where: { email } });
    
    // Si no existe el usuario o la contrasea es incorrecta
    if (!usuario || !(await bcrypt.compare(password, usuario.hash_contrasea))) {
      throw new UnauthorizedError('Email o contrasea incorrectos');
    }
    
    // Generar token JWT
    const token = this.generateToken(usuario);
    
    // Devolver token y datos bsicos del usuario
    return {
      token,
      usuario: {
        id: usuario.id,
        nombre: usuario.nombre,
        email: usuario.email,
      }
    };
  }
  
  // Mtodo para registrar nuevo usuario
  async register(userData: any) {
    // Verificar si ya existe un usuario con ese email
    const existingUser = await this.userRepository.findOne({
      where: { email: userData.email }
    });
    
    if (existingUser) {
      throw new ValidationError([{ field: 'email', message: 'Este email ya est registrado' }]);
    }
    
    // Cifrar la contrasea
    const hashedPassword = await bcrypt.hash(userData.password, 10);
    
    // Crear el nuevo usuario
    const newUser = this.userRepository.create({
      ...userData,
      hash_contrasea: hashedPassword
    });
    
    // Guardar en la base de datos
    const savedUser = await this.userRepository.save(newUser);
    
    // Generar token JWT
    const token = this.generateToken(savedUser);
    
    // Devolver token y datos bsicos del usuario
    return {
      token,
      usuario: {
        id: savedUser.id,
        nombre: savedUser.nombre,
        email: savedUser.email,
      }
    };
  }
  
  // Mtodo privado para generar tokens JWT
  private generateToken(usuario: Usuario) {
    return jwt.sign(
      { id: usuario.id, email: usuario.email, nombre: usuario.nombre },
      process.env.JWT_SECRET!,
      { expiresIn: '24h' }
    );
  }
}

11. Validacin de Datos

Por Qu Validar los Datos

La validacin de datos es crucial para cualquier aplicacin. Imagina que tienes un formulario donde los usuarios deben ingresar su email. Si no validas que realmente sea un email vlido, podras tener problemas ms adelante (por ejemplo, al intentar enviarles un correo).

Los datos de entrada pueden ser invlidos por muchas razones:

  • El usuario cometi un error al escribir
  • Alguien est intentando hacer algo malicioso
  • Hay un bug en el frontend

Por eso, siempre debemos validar los datos antes de procesarlos.

Usando express-validator

En Challenge Plans, usamos express-validator para validar las entradas de usuario:

// controllers/retos.controller.ts con validacin
import { body, param, validationResult } from 'express-validator';

// Definir las reglas de validacin
export const crearRetoValidation = [
  body('titulo')
    .notEmpty().withMessage('El ttulo es obligatorio')
    .isLength({ min: 5, max: 100 }).withMessage('El ttulo debe tener entre 5 y 100 caracteres'),
    
  body('descripcion')
    .optional()
    .isLength({ max: 500 }).withMessage('La descripcin no debe exceder 500 caracteres'),
    
  body('fecha_inicio')
    .notEmpty().withMessage('La fecha de inicio es obligatoria')
    .isISO8601().withMessage('La fecha debe tener un formato vlido (YYYY-MM-DD)'),
    
  body('fecha_fin')
    .notEmpty().withMessage('La fecha de fin es obligatoria')
    .isISO8601().withMessage('La fecha debe tener un formato vlido (YYYY-MM-DD)')
    .custom((value, { req }) => {
      if (new Date(value) <= new Date(req.body.fecha_inicio)) {
        throw new Error('La fecha de fin debe ser posterior a la fecha de inicio');
      }
      return true;
    }),
    
  body('dificultad')
    .isIn(['principiante', 'intermedio', 'avanzado']).withMessage('La dificultad debe ser principiante, intermedio o avanzado'),
    
  // Middleware para verificar los errores
  (req, res, next) => {
    const errors = validationResult(req);
    if (!errors.isEmpty()) {
      return res.status(400).json({ errors: errors.array() });
    }
    next();
  }
];

// Usar las reglas de validacin en la ruta
router.post('/', crearRetoValidation, retosController.crear);

      });
    }

    next();
  } catch (error) {
    next(error);
  }
}

// Usar la validacin personalizada en la ruta
router.put('/:id', authenticate, puedeEditarReto, retosController.actualizar);

14. Glosario: Trminos que Debes Conocer

  • API (Application Programming Interface): Conjunto de reglas que permite a diferentes software comunicarse entre s.

  • REST (Representational State Transfer): Estilo de arquitectura de software para sistemas distribuidos como la World Wide Web.

  • Endpoint: URL especfica donde una API puede acceder a los recursos que necesita.

  • Middleware: Funcin que tiene acceso al objeto de solicitud, al objeto de respuesta y a la siguiente funcin en el ciclo solicitud-respuesta.

  • Router: Parte de Express que permite organizar rutas en archivos o mdulos separados.

  • Controller: Componente que maneja la lgica de las rutas, procesando solicitudes y enviando respuestas.

  • Service: Capa que contiene la lgica de negocio, independiente de la capa de presentacin (controladores).

  • ORM (Object-Relational Mapping): Tcnica de programacin para convertir datos entre sistemas de tipos incompatibles en lenguajes de programacin orientados a objetos.

  • JWT (JSON Web Token): Estndar abierto que define una forma compacta y autnoma de transmitir informacin de forma segura entre las partes como un objeto JSON.

  • Middleware de error: Funcin especial con cuatro argumentos (error, req, res, next) que maneja errores en Express.

  • Autenticacin: Proceso de verificar la identidad de un usuario.

  • Autorizacin: Proceso de determinar si un usuario tiene permiso para acceder a un recurso especfico.
